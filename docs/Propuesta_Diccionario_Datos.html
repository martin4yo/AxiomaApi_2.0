Propuesta: Sistema de Diccionario de Datos

Axioma ERP - Gestión de Metadatos y Documentación Técnica

Problemática Actual

Situación: Actualmente la documentación técnica del sistema (definiciones de tablas, campos, relaciones, reglas de negocio) está dispersa en archivos Word que son difíciles de versionar, buscar y mantener actualizados.

Necesidad: Implementar un sistema profesional de diccionario de datos que sea:

  • Versionable en Git (para control de cambios)
  • Colaborativo (que ambos socios puedan trabajar)
  • Sincronizado con las 2 bases de datos (PostgreSQL + MySQL)
  • Fácil de consultar y buscar
  • Que sirva como fuente de verdad para el desarrollo

Opciones Detalladas

Opción 1: Prisma Schema Enriquecido OPEN SOURCE

Descripción: Aprovechar el schema de Prisma (que ya usan en Node.js) como fuente de verdad, agregando comentarios ricos con metadatos de negocio.

✓ Ventajas

  • Ya están usando Prisma
  • Schema sirve como código Y documentación
  • Genera TypeScript types automáticamente
  • Versionable en Git
  • Prisma Studio para visualización
  • Herramientas de generación de docs

✗ Desventajas

  • Solo cubre PostgreSQL (Node.js)
  • No documenta Django/MySQL directamente
  • Requiere mantener comentarios actualizados

Ejemplo de implementación:

/// Módulos del sistema ERP /// @description Define módulos (Ventas, Compras, Producción, Contabilidad) /// @businessRule Código único alfanumérico por tenant model Modulo { id Int @id @default(autoincrement()) /// Código único del módulo /// @mask "LLNNNN" - 2 letras + 4 números /// @example "VE0001", "CO0001" codigo String @unique @db.VarChar(10) nombre String @db.VarChar(100) // Audit fields createdAt DateTime @default(now()) disabled Boolean @default(false) // Relations entidades ModuloEntidad[] }

Costo: $0 (Open Source)

Opción 2: Django ORM con Docstrings OPEN SOURCE

Descripción: Enriquecer los modelos Django con docstrings detallados y usar django-extensions para generar documentación automática.

✓ Ventajas

  • Ya están usando Django
  • Docstrings en Python son estándar
  • Genera diagramas ER automáticamente
  • Exporta documentación en múltiples formatos
  • Integración con Django Admin

✗ Desventajas

  • Solo cubre MySQL (Django)
  • No documenta Node.js/PostgreSQL
  • Requiere instalar paquetes adicionales

Herramientas complementarias:

django-extensions

Genera diagramas ER y exporta schemas

sphinx-autoapi

Genera docs HTML desde docstrings

drf-spectacular

Ya lo tienen, documenta API REST

Costo: $0 (Open Source)

Opción 3: Dataedo COMERCIAL

Descripción: Herramienta comercial especializada en diccionarios de datos que se conecta directamente a las bases de datos.

✓ Ventajas

  • Diseñado específicamente para esto
  • Se conecta a ambas DBs (PostgreSQL + MySQL)
  • UI profesional y navegable
  • Exporta a PDF, HTML, Excel
  • Búsqueda avanzada
  • No requiere programación

✗ Desventajas

  • Costo anual por usuario (~$500/año)
  • No es "code-first"
  • Versionado limitado
  • Dependencia de herramienta externa
  • No genera código desde docs

Costo: ~$500 USD/año por usuario

Opción 4: SchemaSpy OPEN SOURCE

Descripción: Herramienta Java que genera documentación HTML estática con diagramas interactivos directamente desde las bases de datos.

✓ Ventajas

  • Completamente gratuito
  • Soporta PostgreSQL y MySQL
  • Genera HTML navegable con diagramas
  • Identifica relaciones automáticamente
  • Puede hostearse en GitHub Pages
  • No requiere mantenimiento manual

✗ Desventajas

  • Requiere Java instalado
  • Documentación básica (solo estructura)
  • No incluye reglas de negocio automáticamente
  • UI algo anticuada

Comando de ejemplo:

java -jar schemaspy.jar -t pgsql \ -db axiomadb -host localhost \ -u postgres -p password \ -o docs/postgresql-schema

Costo: $0 (Open Source)

Opción 5: Markdown + MkDocs OPEN SOURCE

Descripción: Crear documentación en archivos Markdown organizados y usar MkDocs para generar un sitio web estático navegable.

✓ Ventajas

  • 100% versionable en Git
  • Fácil de editar (Markdown)
  • MkDocs genera sitio profesional
  • GitHub Pages gratis
  • Búsqueda integrada
  • Totalmente personalizable
  • Soporta diagramas Mermaid

✗ Desventajas

  • Requiere mantenimiento manual
  • No se auto-genera desde DBs
  • Puede desincronizarse con el código
  • Trabajo inicial considerable

Estructura propuesta:

docs/ ├── data-dictionary/ │ ├── general/ │ │ ├── modulo.md │ │ ├── mascara.md │ │ └── persona.md │ ├── entidades/ │ │ ├── entidad.md │ │ └── ejecutivo.md │ └── productos/ │ └── producto.md ├── business-rules/ └── api-reference/

Costo: $0 (Open Source)

Opción 6: Solución Híbrida (RECOMENDADA) RECOMENDADA

Descripción: Combinar lo mejor de todas las opciones: Schema as Code + Documentación generada + Sitio web navegable.

Enfoque integrado:
  1. Enriquecer Prisma schema (PostgreSQL) con comentarios detallados
  2. Agregar docstrings completos en Django models (MySQL)
  3. Script Python que lee ambos schemas y genera Markdown
  4. MkDocs + GitHub Pages para sitio web navegable
  5. CI/CD automático (GitHub Actions) para regenerar al hacer push

✓ Ventajas

  • Documentación sincronizada con código
  • Cubre ambas bases de datos
  • Generación automática
  • Versionado completo en Git
  • Sitio web profesional y navegable
  • Búsqueda avanzada
  • Colaboración fácil (GitHub)
  • CI/CD automático

✗ Desventajas

  • Setup inicial más complejo
  • Requiere escribir script de generación
  • Necesita mantenimiento de comentarios en código

Arquitectura propuesta:

axioma-data-dictionary/ ← Nuevo repositorio ├── .github/workflows/ │ └── deploy.yml ← Auto-deploy a GitHub Pages ├── docs/ │ ├── data-dictionary/ ← Generado automáticamente │ ├── business-rules/ ← Manual │ └── api-reference/ ← Manual ├── schemas/ │ ├── postgresql/ │ │ └── schema.prisma ← Copiado desde project/ │ └── mysql/ │ └── models_export.py ← Copiado desde AxiomaApi_2.0/ ├── scripts/ │ ├── generate-docs.py ← Script de generación │ └── sync-schemas.sh ← Sincroniza desde repos ├── templates/ │ └── table.md.j2 ← Template Jinja2 ├── mkdocs.yml └── README.md

Flujo de trabajo:

1. Developer modifica Prisma schema o Django model 2. Comitea cambios en project/ o AxiomaApi_2.0/ 3. En axioma-data-dictionary/ ejecuta: ./scripts/sync-schemas.sh 4. Script copia schemas y ejecuta generate-docs.py 5. generate-docs.py lee schemas y genera Markdown 6. Git push a axioma-data-dictionary/ 7. GitHub Actions auto-deploy a GitHub Pages 8. Documentación actualizada disponible en URL pública

Costo: $0 (Open Source)

Recomendación Final

Opción recomendada: Solución Híbrida (Opción 6)

Esta opción ofrece el mejor balance entre:

  • Automatización: Generación automática desde código
  • Profesionalismo: Sitio web moderno y navegable
  • Versionado: Todo en Git, historial completo
  • Colaboración: Ambos socios pueden trabajar fácilmente
  • Sincronización: Documenta ambas bases de datos
  • Costo: $0 - Completamente gratis

Plan de Implementación Sugerido

Semana 1: Setup Inicial

  • Crear repositorio axioma-data-dictionary
  • Configurar estructura de carpetas
  • Migrar AXIOMA_ERP_REFERENCE.md como base
  • Configurar MkDocs + GitHub Pages
  • Convertir Word docs a Markdown

Resultado: Sitio web básico funcionando

Semana 2-3: Enriquecimiento de Schemas

  • Agregar comentarios detallados en Prisma schema
  • Agregar docstrings en Django models
  • Desarrollar script de generación (Python)
  • Crear templates Jinja2 para Markdown

Resultado: Generación automática funcionando

Semana 4: Automatización

  • Configurar GitHub Actions para CI/CD
  • Script de sincronización entre repos
  • Validaciones automáticas
  • Documentación de uso para el equipo

Resultado: Sistema completo y automatizado

Inversión de Tiempo Estimada

  • Setup inicial: 8-12 horas
  • Enriquecimiento de schemas: 20-30 horas (una sola vez)
  • Mantenimiento mensual: 2-4 horas (solo actualizar comentarios al modificar código)

Alternativa Rápida (Si tienen poco tiempo)

Si necesitan algo rápido y no pueden invertir tiempo en la solución híbrida:

  • Opción 4 (SchemaSpy): 2-3 horas de setup, auto-genera todo
  • Limitación: Solo estructura, no incluye reglas de negocio
  • Puede combinarse: SchemaSpy para estructura + Markdown para reglas de negocio
